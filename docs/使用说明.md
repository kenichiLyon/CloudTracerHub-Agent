# 云枢助手（CloudTracerHub-agent） 评委使用说明

## 1. 这是一个什么项目

`CloudTracerHub-agent` 是一个面向云原生故障诊断的演示项目，定位是：

- 以 `lubster` 生态形态对外呈现
- 同时提供 `CLI` 与 `MCP Server` 两种使用方式
- 使用固定的五层诊断路径：`现象 -> 资源 -> 服务 -> 依赖 -> 变更`
- 对 Kubernetes、日志、指标、链路和 runbook 做统一证据收集

当前演示版已经内置了可复现的 mock 数据，因此不依赖真实集群，也不依赖真实 Prometheus / Loki / Trace 平台，适合现场演示、录屏和评测。

## 2. 最快体验方式

如果只想在最短时间内看懂项目，建议只执行下面两步：

### 第一步：初始化环境

```bash
uv sync --extra dev
```

### 第二步：直接运行一个演示场景

```bash
uv run cloudtracerhub-agent --config examples/video_demo/lubster.video.config.json --incident-file examples/video_demo/incidents/01_course_selection_timeout.json --format pretty
```

执行后可以看到一份结构化诊断结果，内容会按五层链路展开，并包含：

- 用户输入的故障现象
- Pod / Service / Endpoint 证据
- 日志、指标、链路摘要
- 历史 runbook 检索结果
- 最终建议动作

这也是最推荐的评审入口。

## 3. 建议评审路径

建议按下面的顺序体验，3 到 5 分钟内就能看到项目核心能力。

### 路径 A：CLI 演示（最推荐）

项目已经准备了 5 个演示场景，全部使用同一份 mock 配置，只需要切换 incident 文件。

#### 场景 1：选课高峰接口 502 + Redis 超时

```bash
uv run cloudtracerhub-agent --config examples/video_demo/lubster.video.config.json --incident-file examples/video_demo/incidents/01_course_selection_timeout.json --format pretty
```

建议关注：

- 从 `502 / timeout` 症状一路关联到选课缓存依赖
- 资源、服务、依赖三层证据如何串起来
- runbook 是否能给出“止血 + 回滚/兜底”建议

#### 场景 2：政务材料提交验签失败

```bash
uv run cloudtracerhub-agent --config examples/video_demo/lubster.video.config.json --incident-file examples/video_demo/incidents/02_gov_service_signature_failure.json --format pretty
```

建议关注：

- 故障是否能体现为政务系统里的配置/证书类问题
- 日志里的 `signature verify failed`、`401` 是否能和 trace 证据形成闭环

#### 场景 3：成绩查询链路积压与慢查询

```bash
uv run cloudtracerhub-agent --config examples/video_demo/lubster.video.config.json --incident-file examples/video_demo/incidents/03_grade_query_backlog.json --format pretty
```

建议关注：

- `lag / timeout / slow query` 这类校园系统性能退化问题如何诊断
- 是否能体现“先止血，再定位根因”的工程思路

#### 场景 4：统一身份认证超时

```bash
uv run cloudtracerhub-agent --config examples/video_demo/lubster.video.config.json --incident-file examples/video_demo/incidents/04_unified_auth_timeout.json --format pretty
```

建议关注：

- 教务登录失败如何快速关联到统一身份认证链路
- 会话缓存、认证服务与登录入口之间的依赖关系是否清晰

#### 场景 5：政务预约提交失败

```bash
uv run cloudtracerhub-agent --config examples/video_demo/lubster.video.config.json --incident-file examples/video_demo/incidents/05_gov_appointment_submit_failure.json --format pretty
```

建议关注：

- `503` 与 `mq timeout` 这类异步依赖问题如何体现在五层诊断里
- runbook 是否能给出排队、重试与补偿策略

### 路径 B：MCP 集成演示

如果更关注 Agent / MCP 能力，而不是单纯 CLI，可以直接启动 MCP Server：

```bash
uv run cloudtracerhub-agent-mcp
```

当前主工具名为：

- `lubster_agent_diagnose`

兼容旧工具名：

- `lubster_diagnose`

工具入参示例：

```json
{
  "incident": {
    "title": "选课高峰期间选课接口 502 激增",
    "namespace": "edu-prod",
    "service": "course-gateway",
    "symptoms": ["502", "timeout", "redis timeout"],
    "time_window_minutes": 45,
    "suspect_pod": "course-gateway-6d9c7c8b6d-ptx4q"
  },
  "config_path": "examples/video_demo/lubster.video.config.json"
}
```

MCP 模式适合展示：

- `CloudTracerHub-agent` 可以作为外部客户端可调用的诊断工具，其中 `lubster-agent` 是项目中的诊断能力名
- 同一套五层诊断引擎同时服务 CLI 和 Agent 生态
- 输出同时包含 `structuredContent` 与文本内容，便于继续编排

### 路径 C：Lobster 生态接入

如果更关注“如何挂到 Lobster 生态里”，可以直接查看：

- `lubster_base_config/identity.yml`
- `lubster_base_config/personal.yml`
- `lubster_base_config/workflow.yml`

这三个文件分别定义了：

- Agent 身份：云原生 SRE / 排障专家
- 交互风格：高压故障场景下的沟通与提问方式
- 工作流：五层诊断路径和故障响应方法论

阅读源码时可以这样理解这部分设计：

- `CloudTracerHub-agent` 负责对外提供工程化入口，`lubster-agent` 负责承载具体诊断能力
- `lubster_base_config/` 负责把 Lobster 初始化成符合比赛场景的“云原生排障角色”
- 两者组合后，既有工具能力，也有角色与工作流约束，更容易稳定演示

## 4. 重点可以看什么

这不是一个单点脚本，而是一个可编排的诊断能力封装。建议重点看下面四点：

### 4.1 方法论是否清晰

项目没有直接堆工具，而是把诊断过程固定为：

1. 现象
2. 资源
3. 服务
4. 依赖
5. 变更

这意味着诊断输出不是零散日志，而是可解释、可复用的推理路径。

### 4.2 工具证据是否完整

演示里会统一调用以下证据源：

- `kubectl`
- `log_ql`
- `promql`
- `trace_analyzer`
- `runbook_search`

虽然演示使用 mock 数据，但数据结构和调用方式已经对齐真实工具接口，后续可替换为真实后端。

### 4.3 是否适合作为 Agent 能力节点

项目不仅能本地跑命令，也能通过 MCP 暴露为外部工具，这意味着它可以被更大的 Agent 工作流直接调用。

### 4.4 是否便于稳定评测

评审现场最怕环境不可复现。本项目专门准备了：

- `uv` 管理的可迁移 Python 环境
- 锁定依赖的 `uv.lock`
- 不依赖外部系统的 mock 演示数据

这使得希望使用该项目的使用者即使没有真实 Kubernetes 环境，也能稳定复现同一份结果。

如果需要快速概括这个项目，可以概括为：

- `lubster` 生态下的故障诊断 Agent 封装
- 兼容 `CLI + MCP` 双入口
- 五层诊断链路清晰
- 支持可复现的离线演示
- 适合作为更大 Agent 系统中的诊断节点
